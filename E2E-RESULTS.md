## Defects found + fixed

1. **`?next=` deep-link preservation dropped on session-expiry redirects.** `src/app/(app)/layout.tsx`'s auth guard called bare `redirect("/login")`, losing the originally-requested path (login/join's own `?next=` handling was already correct — only this one call site was broken). **Fix**: middleware (`src/lib/supabase/middleware.ts`) now forwards the current path via an `x-pathname` request header; the layout reads it via `headers()` and rebuilds `?next=`. Verified: `/circles/new` while logged out now correctly round-trips through magic-link auth back to `/circles/new` instead of dropping to `/home`.
2. **Join-via-link button could get stuck on "Joining…" forever, silently never joining.** `src/components/entry/join-button.tsx`'s `onClick` synchronously set React state that drove the button's `disabled` attribute — a `<button disabled>` cannot submit its own form, so on some click timings the button disabled itself before the browser's native form submission fired, and the tap did nothing (verified: user row created via login, but no `circle_members` row, no server-side POST logged, reproduced twice). This breaks HANDOFF's explicitly named "#1 growth flow." **Fix**: switched to `aria-disabled` + `pointer-events-none` (visual-only), which can't race the native submit. Verified fixed twice (Ivy, then Jack) with the DB row actually appearing and landing on the Circle feed. A parallel-agent code review confirmed no other submit button in the codebase has the same pattern.
3. **RSVP confirmed-slot order was non-deterministic**, violating "slots fill in order." `getSessionSummary` in `src/server/games-service.ts` never sorted the `confirmed` array (only the reserve queue had an explicit `.sort()` by position); `rsvpIn()` never assigns "in" rows a `position` either, so display order fell back to arbitrary DB row order — observed as newest-RSVP-first instead of first-come-first-served. **Fix**: sort `confirmed` by the already-tracked `respondedAt` timestamp. Verified: Bob/Cara/Dave/Grace now render left-to-right in RSVP order, matching the order they tapped in.
4. **Fourth Call ring 1 always showed a premature green checkmark**, even 6 days before the T-48h window opens, contradicting the design prototype's data-driven ring-1 icon (confirmed against `CUATRO-Prototype.dc.html`'s `{{r1Icon}}`/`{{r1Bg}}` template vars). `fourth-call-send.tsx` hardcoded ring 1's icon to `bg-win`/"✓" regardless of state, unlike ring 2 which correctly branches on `ring2State`. **Fix**: added a computed `ring1State` (mirroring ring 2's pending/sent logic) and made ring 1's icon conditional on it. Verified: a session 6 days out now correctly shows ring 1 as pending ("1" outline icon) until it actually fires.
5. **Site-wide: every `<Link>`/`<a>`'s `text-*` colour utility was being silently overridden to the ambient inherited colour.** `globals.css`'s `a { color: inherit; }` (and the rest of the base reset — `*`, `html/body`, `button`, `input`) was placed *outside* any `@layer` block, right after `@import "tailwindcss"`. Per CSS cascade-layers semantics, unlayered rules always beat rules inside a named layer (Tailwind's utilities live in `@layer utilities`) regardless of selector specificity — so `a { color: inherit }` beat every `.text-action`/`.text-strong-fg`/etc. applied to a link, app-wide. Found via the "Log last night's result" CTA rendering with **fully invisible text** (`bg-strong-bg` and `text-strong-fg` resolved to the same colour because `text-strong-fg`'s intended dark colour was being overridden back to the ambient light ink colour, which happened to equal the light `bg-strong-bg`). Also silently broke the "‹ Games" back link's intended coral colour (rendered as plain ink instead). **Fix**: wrapped the whole base-reset block in `@layer base { ... }`, restoring normal Tailwind precedence. Verified: the CTA text is now fully readable, and the back-link now renders coral as intended. This is a meaningful, previously-invisible, app-wide defect — worth flagging for a broader visual sweep post-fix since it likely affected other links not directly exercised in this pass.

All four fixes are minimal, targeted, and the full test suite (238/238) stayed green after each one; no test needed updating since none of the bugs were covered by existing tests (all four were pure UI/wiring issues invisible to the unit-test layer, which is exactly why an actual browser pass caught them).

## Escalations (not re-verified live; judged safe based on code + existing test coverage)

1. **Chat reconnect catch-up** — `backfill()` on the synthetic `reconnect` event is implemented and reads correctly, but wasn't re-tested via an actual network kill/restore in the browser (no easy way to simulate offline in this environment without disrupting other in-flight tests).
2. **Late-cancel vs early-cancel reliability distinction** — logic exists and is unit-tested (`LATE_CANCEL_WINDOW_MS` gate), not independently re-driven through a live RSVP-out at each side of the boundary.
3. **Fourth Call L2's exact "shared-circle/opponent-history ±0.5 band" candidate matching** — the escalation mechanism and manual "Escalate now" trigger were verified live, but the candidate-selection algorithm itself wasn't exercised with a genuine extended-network user (would require constructing shared-circle or match-history state for a brand-new test user; used ring-3's public-link claim instead, which is functionally the same "claim without joining" outcome the charter asks for).
4. **Retired match visual flow** — verified via code (UI badge exists) and an existing passing unit test (zero-games, no rating movement), not re-driven through the live "Match was retired early" checkbox end-to-end.

None of these are believed to be broken — they're time-boxing decisions given the volume of ground covered, backed by either direct code reading or existing automated coverage.
